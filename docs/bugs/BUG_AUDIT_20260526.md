## 🔴 BUG B01 (FALSO POSITIVO) — quote_items subtotal

**Tabela:** `quote_items`  
**ID:** `ed95740c-370d-48d4-8187-33353c49a166`  
**Cotação:** `10001/26`  

Análise inicial detectou divergência. Investigação profunda revelou que o valor é **correto**:

```
unit_price = 21.50
quantity   = 200
base       = 21.50 × 200 = 4.300,00
personalization_cost = 1.087,90
subtotal correto = 4.300,00 + 1.087,90 = 5.387,90 ✅
```

A query de auditoria usava `subtotal != unit_price * qty` sem considerar `personalization_cost`.  
**Ação:** CHECK CONSTRAINT `chk_quote_items_subtotal_positive` adicionado para garantir `subtotal >= 0`.

## 🔴 BUG B02 — `negotiation_markup_percent` não persistido

**Tabela:** `quotes`  
**Registros afetados:** 3 cotações (ORC-2026-001, ORC-2026-002, ORC-2026-003)

### Evidência

| Cotação | Subtotal | Total | Markup Real | Campo |
|---|---|---|---|---|
| ORC-2026-001 | R$5.000 | R$5.250 | 5,00% | 0,00% ❌ |
| ORC-2026-002 | R$12.500 | R$13.125 | 5,00% | 0,00% ❌ |
| ORC-2026-003 | R$800 | R$840 | 5,00% | 0,00% ❌ |

O front-end aplica o markup no `total`, mas o campo `negotiation_markup_percent` não é persistido no upsert.

### Fix Aplicado
Backfill dos 3 registros:
```sql
UPDATE quotes SET negotiation_markup_percent = ROUND((total/subtotal - 1) * 100, 4)
WHERE negotiation_markup_percent = 0 AND total > subtotal AND ...
```

### Fix Pendente no Código
Investigar o hook/RPC responsável por salvar cotações. O payload do `upsert` deve incluir `negotiation_markup_percent` sempre que `total != subtotal`.

**Arquivos suspeitos:**
- `src/hooks/useQuotes.ts`
- `src/components/admin/` — formulários de cotação
- Edge function `quote-sync` ou RPC `fn_quotes_recalc_subtotal`

## 🔵 ACHADO B05 — 136 Produtos com `category_id ≠ main_category_id`

Não é um bug de dados — é uma inconsistência de **semântica de campo**.

| Campo | Significado | Exemplo |
|---|---|---|
| `category_id` | Subcategoria folha (específica) | "Canetas \| Plástico" |
| `main_category_id` | Categoria raiz (navegação) | "Canetas" |

Para 136 produtos, os dois campos apontam para níveis diferentes na hierarquia — o que é **correto e esperado** para produtos importados com subcategorias específicas.

**Recomendação:** Documentar formalmente no schema e garantir que as queries de busca/filtro usem o campo correto para cada propósito.
